## Overview

You can create up to **5 named gateway API keys** from **Dashboard → API Keys**. Naming keys by workflow or machine (e.g., `Laptop`, `Desktop`, `Remote Server`, `CI`) makes it easy to track which environment is consuming capacity and revoke a single key without disrupting others.

Keys are shown in full only at creation time. After that, the dashboard masks the secret and displays a `last_used_at` timestamp so you can audit live connectivity.

## Key format

All gateway keys use the `csk_` prefix:

| Prefix | Use |
|--------|-----|
| `csk_...` | Live subscription key for IDE agents and API calls |
| `csk_test_...` | Test key for development (limited capacity) |

Use `csk_...` keys for all OpenAI-compatible integrations. Pass them as `Authorization: Bearer csk_...`.

## Public Runtime Usage

```bash
CENCORI_API_KEY=csk_...
```

```bash
curl https://api.cencori.com/v1/models \
  -H "Authorization: Bearer $CENCORI_API_KEY"
```

For native Cencori SDK/REST calls, `CENCORI_API_KEY: csk_...` is also accepted.

## Programmatic Key Management

Public API-key management endpoints are not available yet. Dashboard routes under `/api/projects/{projectId}/api-keys` require a signed-in Cencori owner session and should not be used from customer product code.

## Best Practices
- Rotate keys regularly (every 90 days recommended).
- Use separate keys for different environments.
- Revoke unused keys immediately.
- Never commit keys to version control.
- Never expose a `csk_...` key in browser code or a mobile app bundle.
