## Credits

Cencori Code abstracts away raw per-token costs through an internal unit called a **Credit**.

```
1 Credit = $0.001
```

This fixed mapping means your subscription plan stays stable even when Anthropic, OpenAI, or DeepSeek adjust their upstream pricing. Cencori absorbs those changes behind the scenes without modifying your plan.

Credit balances are never shown as raw numbers in the public dashboard. Usage is displayed as simple progress bars across your rolling windows so you always know how much headroom you have left — without staring at a fluctuating dollar meter mid-workflow.

## The three-tier rolling window engine

Every inbound request must clear three separate rolling windows simultaneously before it is dispatched. This architecture protects you from runaway IDE agent loops and protects the platform from sudden financial exposure.

### 5-Hour Rolling Window

A short-term throttle. It catches and stops broken automated background scripts or looping agents before they cause severe damage within a single work session.

### 7-Day Rolling Window

A mid-term bucket. It manages sustained delivery across an engineering week. If a long refactoring sprint is chewing through capacity, this window flags it before it compounds over the month.

### Monthly Billing Window

A hard cap aligned to your Paystack renewal date. When this window resets, your full monthly allocation is restored.

## Request lifecycle

Each request follows a strict 5-step path:

**1. Authorization** — the bearer token is authenticated against the cache layer and matched to your active tier.

**2. Worst-case estimation** — before touching any upstream provider, the proxy calculates the maximum possible credit cost:

```
Cost_max = (Tokens_input × Rate_input) + (Tokens_max_out × Rate_output)
```

`Tokens_max_out` is taken from your request's `max_tokens` parameter.

**3. Atomic reservation** — `Cost_max` is atomically deducted from all three rolling window counters at once. This reservation blocks concurrent microsecond-level racing requests from bypassing the caps.

**4. Upstream streaming** — the request is forwarded. The response stream is piped back to your IDE without server-side buffering, keeping added latency under **p95 ≤ 30ms**.

**5. Settlement** — once the stream completes, the actual cost is calculated from real token counts:

```
Cost_actual = (Tokens_input × Rate_input) + (Tokens_actual_out × Rate_output)
```

The difference `Cost_max − Cost_actual` is immediately credited back to your rolling windows. An immutable ledger record is written asynchronously for audit purposes.

---

## Rate limit responses

If any single rolling window lacks the required headroom, the proxy blocks the request immediately:

```
HTTP 429 Too Many Requests
```

The response body identifies the specific window that was exhausted and its calculated reset time. No upstream request is made, so no credits are consumed.

## Billing

Subscriptions are processed through **Paystack**, with native support for:
- Nigerian Naira (NGN)
- Global credit cards
- Instant bank-transfer networks

Successful payments trigger an immediate monthly capacity reset. Failed recurring collections or cancellations gracefully step down access at the natural end of the paid billing cycle — no abrupt cutoff mid-session.
